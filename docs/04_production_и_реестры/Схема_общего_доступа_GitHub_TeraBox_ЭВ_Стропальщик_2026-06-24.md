## Назначение

Этот документ задает рабочую схему общего доступа к материалам проекта для:

- обмена документацией;
- проверки совместимости работы между коллегами;
- передачи файлов в финальную сборку `PowerPoint + iSpring`;
- хранения тяжелых фото, видео, `pptx`, экспортов и сопутствующих материалов.

Главная идея:

- `GitHub` используем как источник структуры, логики, документации, реестров, скриптов и легких рабочих файлов;
- `TeraBox` используем как общее файловое хранилище для тяжелых медиа, рабочих пакетов `PowerPoint/iSpring` и проверочных сборок.

## Почему схема делится на два слоя

### `GitHub`

Храним в репозитории:

- все `.md`, `.csv`, `.ps1`, `.py`;
- рабочие реестры и production-документы;
- шаблоны, инструкции, логику навигации;
- легкие превью и контрольные сборки, если они реально нужны в истории проекта.

Не рассчитываем на GitHub как на основное хранилище тяжелых исходников фото и видео.

### `TeraBox`

Храним в TeraBox:

- тяжелые фото и видео;
- исходники для монтажа и анимации;
- рабочие `pptx` и проверочные `ppsx`, которые коллеги открывают руками;
- материалы, которые нужны сборщику `iSpring`;
- контрольные пакеты для сверки совместимости.

## Критичное правило по путям

Для `TeraBox`-пакетов и любых файлов, которые должны открываться в `PowerPoint` или использоваться в `iSpring`, используем только `ASCII`-имена:

- латиница;
- цифры;
- дефис `-`;
- нижнее подчеркивание `_`.

Не используем в обменном слое:

- кириллицу в именах файлов и папок;
- пробелы;
- слова вроде `final`, `final2`, `последний`, `новый`.

Причина:

- в проекте уже подтвержден риск проблем с открытием `PowerPoint`-файлов при кириллических путях;
- для обменного контура нужна максимально переносимая структура.

Для конечной сборки должны быть доступны одновременно:

- рабочие `pptx`;

- видео;
- изображения;

- информация о переходах и возвратах.

Минимальные обязательные блоки:

1. `структура курса`
2. `рабочий powerpoint-файл`
3. `review-сборка`
4. `media для модуля`
5. `реестр комплектности`
6. `реестр навигации`

## Единое правило именования пакетов обмена

Для всех новых пакетов общего доступа используем одну схему:

`YYYY-MM-DD_module-XX_package-type[_vNN]`

Где:

- `YYYY-MM-DD` — дата сборки пакета;
- `module-XX` — модуль или рабочая область, например `module-01`;
- `package-type` — назначение пакета;
- `_vNN` — версия, если в один день собираем повторный пакет того же типа.

### Разрешенные значения `package-type`

- `sync` — пакет сверки совместимости;
- `review` — пакет для просмотра и проверки переходов;
- `ispring-handoff` — комплект для финальной сборки в `iSpring`;
- `media-refresh` — обновление медиа без пересборки всей логики;
- `archive-copy` — архивная копия перед переносом в `08_archive`.

### Как именуем папку и zip

Имя внутренней папки и имя архива должны совпадать по базовой части.

Пример:

- папка: `2026-06-24_module-01_sync`
- архив: `2026-06-24_module-01_sync.zip`

Если в тот же день выпускаем повторную сборку:

- папка: `2026-06-24_module-01_sync_v02`
- архив: `2026-06-24_module-01_sync_v02.zip`

### Базовые примеры для проекта

- `2026-06-24_module-01_sync.zip`
- `2026-06-26_module-01_review_v02.zip`
- `2026-06-28_module-01_ispring-handoff.zip`
- `2026-07-02_module-02_sync.zip`

### Правила версионирования

- если меняется содержимое пакета в тот же день, повышаем только версию: `v02`, `v03`, `v04`;
- если пакет собирается в другой день, дата меняется обязательно;
- если меняется назначение пакета, меняется и `package-type`;
- не используем слова `final`, `new`, `latest`, `последний`, `исправленный`.

### Что считаем новым пакетом обмена

Новый пакет собираем, если произошло хотя бы одно из изменений:

- обновился рабочий `pptx`;
- обновился проверочный `ppsx`;
- добавились обязательные медиа для сборки;
- изменилась навигация или логика переходов;
- пакет передается новому участнику или в новый этап сборки.

## Роли файлов по типам

### `working`

Редактируемый файл.

Пример:

- `S001-S007_live_preview_working_2026-06-24_v32.pptx`

### `review`

Файл только для просмотра и кликов.

Пример:

- `S001-S007_review_clickable_2026-06-24_v02.ppsx`

### `build-input`

Файл, который нужен сборщику для конечной интеграции в `iSpring`.

Примеры:

- аудио;
- видео;
- экспортированные картинки;
- финальные `pptx`.

## Именование файлов в общем обменном слое

Для `TeraBox` используем шаблон:

`<package-or-slide-id>_<meaning>_<date>_vNN.<ext>`

Примеры:

- `S001-S007_live_preview_working_2026-06-24_v32.pptx`
- `S001-S007_review_clickable_2026-06-24_v02.ppsx`
- `module-01_media_required_2026-06-24_v01.zip`
- `ev-strop_manifest_2026-06-24_v01.csv`

## Что нельзя выкладывать в общий пакет

Не включаем:

- `.git/`
- `temp/`
- `__pycache__/`
- случайные тестовые `pptx/ppsx`
- дубли одной и той же версии без роли
- файлы без понятного назначения

## Правило версий

### Для рабочих `pptx`

- версия повышается при каждом зафиксированном ручном изменении в PowerPoint;
- финальная на сегодня версия должна быть одна.

### Для `review .ppsx`

- версия повышается только после выпуска из соответствующего рабочего `pptx`;
- `review` всегда должен ссылаться на конкретный `working`.

### Для общего пакета

- одна дата;
- один номер версии;
- один манифест;
- один ответственный.

## Минимальный алгоритм публикации

1. Зафиксировать рабочий `pptx`.
2. Выпустить из него `review .ppsx`.
3. Обновить манифест.
4. Собрать папку `07_compatibility_check/<date>_module-01_sync/`.
5. Проверить, что все имена в пакете ASCII.
6. Загрузить пакет в `TeraBox`.
7. Отдать коллегам ссылку на пакет и отдельную ссылку на манифест.
8. Попросить коллег проверять совместимость только по этой версии пакета.

## Практический вывод

Для проекта `ЭВ Стропальщик` правильная схема сейчас такая:

- `GitHub` — логика, документы, реестры, скрипты;
- `TeraBox` — тяжелые материалы и обменные сборочные пакеты;
- `PowerPoint working` и `review ppsx` живут как разные роли;
- для коллег публикуем не весь проект, а `контролируемый compatibility-пакет`.
