# README: система управления изменениями

Дата фиксации: `2026-06-20`

## Назначение

Этот документ нужен как короткая навигация по системе управления изменениями в проекте `ЭВ Стропальщик`.

Он отвечает на три вопроса:

1. Какой документ открывать первым.
2. Какой документ использовать для какого типа изменения.
3. В какой последовательности фиксировать изменение, чтобы не потерять структуру, нумерацию, тесты и production-связки.

## Главный принцип

В проекте разделяем два разных слоя изменений:

- `изменение_структуры` — меняется состав или порядок линейного маршрута `S001-S056`;
- `изменение нумерации` — меняются `S / P / PP / P56`, возвраты, дочерние связи и связанные `ID`.

Если изменение затрагивает оба слоя, его нужно проводить по обоим контурам.

## Какая цепочка считается правильной

### Если меняется структура курса

1. Оформить запрос:
   `docs/00_навигация_и_изменения/Шаблон_запроса_на_структурное_изменение_курса_2026-06-20.md`
2. Зафиксировать решение:
   `docs/00_навигация_и_изменения/Шаблон_решения_по_структурному_изменению_курса_2026-06-20.md`
3. Обновить рабочие документы структуры:
   утвержденную структуру, master-таблицу, production-реестр
4. Внести запись в:
   `docs/00_навигация_и_изменения/Журнал_изменений_структуры_курса_S001_S056_2026-06-20.md`
5. Если изменение потянуло за собой новые `ID`, отдельно пройти контур нумерации.

### Если меняется только нумерация

1. Проверить правило в:
   `docs/01_структура_курса/Схема_нумерации_слайдов_основной_подвал_подподвал_2026-06-18.md`
2. Использовать рабочий шаблон:
   `docs/00_навигация_и_изменения/Шаблон_таблицы_изменений_нумерации_слайдов_2026-06-20.md`
3. Обновить master-уровень, дочерние `P / PP / P56`, возвраты, тестовые узлы, файлы и реестры
4. Внести запись в:
   `docs/00_навигация_и_изменения/Журнал_изменений_нумерации_слайдов_2026-06-20.md`

## Минимальные правила проекта

- нельзя менять `ID` только в одном документе и считать работу завершенной;
- нельзя проводить структурную правку без записи в соответствующий журнал;
- если изменение затронуло тестовый узел, его нужно проверять как отдельный мини-сценарий;
- перед сборкой `PowerPoint/iSpring` рабочую версию структуры и нумерации лучше замораживать пакетами.

## Практический вывод

Система считается полной только в такой связке:

`правило -> запрос -> решение -> правка -> журнал структуры -> журнал нумерации -> production-проверка`

Если один из этих шагов пропущен, риск расхождения между структурой, нумерацией, тестами и файлами резко растет.
